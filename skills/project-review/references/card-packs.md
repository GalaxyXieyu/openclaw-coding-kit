# Project Review Card Packs

## 设计原则

- 卡片类型固定，不做通用卡片引擎
- 一个卡片只解决一个场景
- 默认先让用户“一眼看懂”，再考虑花样
- 所有项目摘要都走同一套文案规则

## `weekly_review_card_v1`

### 使用场景

- 每周固定推送到项目群
- 适合多项目并排展示

### 信息结构

1. 标题：`本周项目回顾`
2. 摘要区：每个项目一行
3. 总结区：本周共完成多少项、还压着多少项
4. 下一步区：下周优先看什么

### 每个项目的呈现

- 项目名
- 50 字内一句话摘要
- 可选状态标记：`推进中` / `有阻塞` / `待确认`

### 推荐动作

- `打开任务列表`
- `本周归档`
- `三天后再提醒`

### 点击后的归档

- `打开任务列表`: 不改 review 状态，只记录点击
- `本周归档`: 状态改为 `archived`
- `三天后再提醒`: 状态改为 `snoozed`

## `monthly_review_card_v1`

### 使用场景

- 每月固定推送到项目群
- 更适合看趋势和取舍

### 信息结构

1. 标题：`本月项目复盘`
2. 项目摘要区：每个项目一行
3. 趋势区：这个月推进最多 / 拖得最久的项目
4. 下月建议区：建议先做的 1-3 件事

### 推荐动作

- `打开任务列表`
- `查看本月详情`
- `本月归档`

### 点击后的归档

- `查看本月详情`: 可跳到 PM doc 或 review 明细页
- `本月归档`: 状态改为 `archived`

## `event_alert_card_v1`

### 使用场景

- 重要节点触发
- 一次只说一件事

### 事件类型

- `stale_project`
- `blocked_project`
- `important_done`
- `too_many_pending`

### 信息结构

1. 标题：`项目提醒`
2. 事件一句话：说清楚发生了什么
3. 建议动作：明确下一步
4. 证据区：最多两条任务事实

### 推荐动作

- `去看任务`
- `今天处理`
- `一周后再提醒`
- `这次不用提醒`

### 点击后的归档

- `今天处理`: 状态改为 `acked`
- `一周后再提醒`: 状态改为 `snoozed`
- `这次不用提醒`: 状态改为 `archived`

## `code_health_risk_card_v1`

### 使用场景

- 最近 24 小时内有提交
- 需要做一次自动代码体检

### 信息结构

1. 标题：`代码健康提醒`
2. 风险概览：`P0 / P1 / P2` 数量
3. 重点风险：最多 3 条
4. 文档治理：docs 漂移、过期文件候选、`AGENTS.md` 漂移
5. 修复影响：是否需要补 `ui-ux-review`

### 推荐动作

- `开始修复`
- `查看详情`
- `明天再看`
- `忽略这次`

### 点击后的归档

- `开始修复`: 状态改为 `acked`，并触发修复流
- `查看详情`: 保持当前状态，只记录点击
- `明天再看`: 状态改为 `snoozed`
- `忽略这次`: 状态改为 `archived`

## 通用字段建议

每张卡都建议带上这些内部字段：

- `review_id`
- `card_kind`
- `project_keys`
- `source_task_ids`
- `dedupe_key`
- `sent_at`
- `requires_uiux`
- `fix_scope`

## 幂等规则

- 同一个 `dedupe_key` 只允许发送一次有效卡片
- 如果只是更新点击状态，应更新原卡片，不重复发新卡
- 如果 `snooze_until` 未到，不重新触发同类提醒

## 后续扩展

第二阶段如果要加入知识沉淀，再新增卡片类型，不直接改写现有三类卡片的职责。
