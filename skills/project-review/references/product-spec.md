# Project Review Product Spec

## 目标

`project-review` 的目标不是再做一个任务系统，而是在现有 `pm` 之上补一层主动复盘与推送能力，让项目管理从“要你自己去看”变成“系统主动告诉你现在到哪了”。

它也不是几个平级 review skill 的集合页，而是一个统一入口。日常项目复盘、代码 review、文档治理、UI/UX 验证，都应收口到 `project-review` 下面。

第一阶段只做三类主动输出：

- 周报
- 月报
- 关键事件提醒
- 代码健康风险卡

## 双轨模型

`project-review` 现在可以理解成一个 umbrella skill，下有几条 review 线：

- **项目复盘线**：看任务推进、阻塞、下一步
- **代码健康线**：看最近提交后的代码风险、文档漂移、治理缺口
- **验证线**：在修复后补 UI/UX 验证
- **结构观察线**：周/月级图谱观察，不参与每日必跑

前者解决“项目现在到哪了”，后者解决“仓库会不会越写越乱”。

## 用户故事

### 1. 项目周报

作为项目负责人，我希望每周在项目对应的飞书群里收到一条简短周报，这条周报不用点开很多页面就能看懂每个项目这周做了什么、还差什么、下周先干什么。

成功标准：

- 一条卡片能覆盖多个项目
- 每个项目一行大白话摘要
- 摘要尽量控制在 50 字内
- 点击后能跳到任务列表或标记已读

### 2. 项目月报

作为项目负责人，我希望每月收到一条更完整的月报，用来判断哪些项目在推进，哪些项目拖住了，下一阶段优先级要怎么排。

成功标准：

- 比周报多一点趋势信息
- 仍然以简单语言为主
- 不出现行业黑话和专业套话
- 能给出下一步建议

### 3. 关键事件提醒

作为项目负责人，我希望系统在值得提醒的时候主动推送，而不是每天机械打卡。

触发示例：

- 某项目连续多天没有更新
- 某重要任务刚完成，适合发一个阶段提醒
- 某项目堆积待办过多，需要做取舍
- 某阻塞超过阈值还未处理

### 4. 代码健康巡检

作为长期项目维护者，我希望如果前一天有代码提交，系统能自动做一轮代码健康巡检，把高风险问题、文档问题和治理问题汇总成一张风险卡。

成功标准：

- 只在有新提交时触发
- 先看 Git 变更，再补仓库治理检查
- 输出 P0 / P1 / P2 风险
- 如果点击“修复”，系统会进入自动修复流
- 如果修复涉及 UI，再补一轮定向 `ui-ux-review`

## 系统边界

### `pm` 负责

- 任务和文档真相
- 当前任务、待做任务、已完成任务的快照
- 评论和完成写回

### `project-review` 负责

- 复盘输入整理
- 文案生成
- 卡片结构
- 发送去重
- 点击回调
- 归档状态
- 代码健康巡检编排
- docs / AGENTS 治理检查
- UI/UX 验证编排
- 图谱观察编排

### Feishu delivery 层负责

- 实际发消息
- 实际更新卡片
- 打开任务链接

## 推荐输入模型

最小输入对象可以抽象成：

```json
{
  "project": {
    "name": "小程序",
    "channel_id": "oc_xxx"
  },
  "trigger": {
    "kind": "weekly",
    "window_start": "2026-04-07T00:00:00+08:00",
    "window_end": "2026-04-14T00:00:00+08:00"
  },
  "tasks": {
    "completed": [],
    "active": [],
    "blocked": [],
    "stale": []
  }
}
```

`pm` 给出的是原始任务事实，`project-review` 再把这些事实压缩成“人能一眼看懂”的输出。

## 产出模型

### `weekly_review_card_v1`

- 面向项目群
- 一张卡片展示多个项目
- 每个项目一句摘要
- 附一个总的“下周先看什么”

### `monthly_review_card_v1`

- 面向项目群
- 更强调趋势和取舍
- 展示本月推进、未完成重点、下月先做什么

### `event_alert_card_v1`

- 面向项目群或指定接收人
- 一张卡只说一件事
- 必须带动作建议

### `code_health_risk_card_v1`

- 面向项目负责人或维护群
- 一张卡展示最近提交带来的主要风险
- 必须区分 P0 / P1 / P2
- 必须能点击“修复”或“稍后处理”

## 推送频率建议

默认推荐值，不作为硬编码常量：

- 周报：每周最后一个工作日傍晚
- 月报：每月最后一个工作日傍晚
- 事件提醒：规则触发后即时发送，但 24 小时内同类提醒要去重

## 状态机

每个 review 事件至少经历以下状态：

1. `detected`
2. `drafted`
3. `sent`
4. `acked` / `snoozed` / `archived`

如果用户点击“稍后提醒”，状态进入 `snoozed`，同时记录 `snooze_until`。
如果用户点击“已读归档”或“本周归档”，状态进入 `archived`。

## 写回规则

### 卡片发送时

- 记录 review_id
- 记录 dedupe_key
- 记录消息目标和发送时间

### 用户点击时

- 先更新 review state
- 再按动作决定是否写回 `pm`

建议动作和写回：

- `open_tasks`: 只跳转，不写回任务
- `acknowledge`: 写一条简短 PM comment，表示已读
- `snooze_3d`: 写 review state，不写任务状态
- `archive`: 写 review archive 记录，必要时补一条 PM comment
- `fix_now`: 创建或激活一个修复任务，再触发自动修复流

## 设计取舍

### 为什么不先做知识卡

因为当前最大的痛点不是缺少卡片，而是缺少主动提醒和压缩总结。先把“主动推送”做稳，才能让后续知识沉淀有固定入口。

### 为什么不依赖 `feishu-card-os`

因为本项目现在需要的是稳定、窄范围、可控的 review 能力，不需要一套通用卡片平台。可以借鉴其状态机和幂等思路，但不直接耦合实现。

### 为什么 code review 不能只做 diff review

因为长期项目变脏的原因不只有“这次代码写得对不对”，还包括：

- 文件越来越长
- 重复代码慢慢堆积
- 文档没更新
- 过期说明文件还留着
- `AGENTS.md` 和真实流程脱节

所以需要一条“最近提交触发 + 仓库治理补扫”的组合流程。

## MVP 验收口径

第一阶段完成，至少要满足：

- 可以从 `pm` 任务快照生成周报/ 月报/ 事件提醒三类结构
- 可以在最近 24 小时有提交时生成代码健康风险卡
- 每个项目都能产出一条 50 字内的大白话摘要
- 卡片点击后有明确归档和回写语义
- 同一个提醒不会反复轰炸
