# Project Review Code Health Spec

## 目标

代码健康巡检不是每天机械跑全量审查，而是在“最近一天有提交”这个前提下，做一轮有边界的仓库体检，然后把结果压缩成一张可以点击处理的风险卡。

这条能力应作为 `project-review` 的内部 lane 来运行，而不是拆成多个平级 skill 再手工拼起来。

## 触发规则

默认规则：

1. 检查最近 24 小时是否有新的 Git commit
2. 没有提交就不触发
3. 有提交才进入代码健康巡检

推荐使用 `scripts/commit_window.py` 做第一层检测。

## 巡检流程

### Phase 1: 提交窗口检测

- 读取最近 24 小时提交
- 合并 commit 范围
- 提取 changed files

### Phase 2: `code-review` lane

这里吸收 `code-review` 的核心思路，但在 `project-review` 里内化执行：

- 保留 `P0 / P1 / P2`
- 保留“改什么 / 怎么改 / 为什么”
- 保留按 Git 变更动态加载标准

需要吸收的检查重点：

- 过长函数和过长文件
- 重复代码没有复用
- 高风险改动缺少测试
- API / 状态码 / 错误处理不一致
- 性能和资源使用问题

## Phase 3: `docs-review` lane

这部分是 daily quality retro 必须补上的治理检查。

建议至少检查：

- `docs` 漂移：代码改了，说明没跟
- 过期文档候选：明显失效的说明文件、废弃流程说明
- `AGENTS.md` 漂移：仓库约束和真实流程不一致
- 目录杂物：临时导出、旧报告、重复草稿长期堆积
- 重复工具：功能近似但分散在多个脚本

注意：

- “删除过期文件”不要直接自动执行
- 第一阶段只出候选清单和风险理由
- 真删除要走用户点击或明确修复流

## Phase 4: 风险分级

### P0

- 会导致线上故障、数据风险、严重回归
- UI 主路径损坏
- 关键文档和真实行为完全相反

### P1

- 会明显降低维护效率
- 重复代码持续扩散
- 重要文档长期未更新
- `AGENTS.md` 已经误导后续代理

### P2

- 风格不统一
- 文件偏长但暂未形成实质风险
- 可删可不删的旧文件候选

## 风险卡输出

风险卡至少包含：

- 本次 commit 时间窗
- 影响文件数
- P0 / P1 / P2 数量
- 重点风险 Top 3
- docs / AGENTS / 过期文件候选
- 是否需要 UI/UX 验证

## 点击“开始修复”后的动作

推荐修复流：

1. 创建或激活一个 PM 修复任务
2. 先修 P0，再修 P1
3. 如果涉及文档漂移，同时更新 docs 和 `AGENTS.md`
4. 如果修复影响 UI 页面、组件、样式、交互：
   - 自动触发定向 `ui-ux-review` lane
   - 不是全量跑整个项目，而是按受影响模块做 smoke + 证据回填
5. 修复完成后回写 PM comment 和 review archive

## `ui-ux-review` lane 的接入边界

这里参考 `ui-ux-test` 的方法，但运行时仍归 `project-review` 调度。

建议规则：

- 只有当改动命中 UI 文件时才触发
- 优先做定向 smoke
- 失败结果写成证据，不直接自动判定为产品问题

常见 UI 命中信号：

- `pages/`
- `components/`
- `src/app/`
- `src/pages/`
- `*.wxml`
- `*.wxss`
- `*.tsx`
- `*.jsx`
- `*.css`
- `*.scss`

## `graph-observe` lane 的角色

这里可以参考 `graphify`，把它作为结构观察 lane，而不是每日主巡检。

适合的用法：

- 每周或每月增量更新一次图谱
- 用 `GRAPH_REPORT.md` 看结构热点和孤立区域
- 用 `graph.html` 给人看整体关系
- 用 query/path 能力追查某类文件为何没有复用

不建议：

- 每天提交后都跑一次全量图谱
- 让图谱工具直接改写本仓库的 `AGENTS.md`

原因：

- 图谱更适合做“结构观察”，不是“快速修复”
- 图谱安装动作可能会改 `AGENTS.md` 和 hooks，可能和现有 PM/AGENTS 约束冲突

## 推荐 MVP

第一阶段把代码健康线做到下面这个程度就够了：

1. 最近 24 小时 commit 检测
2. `code-review` lane
3. `docs-review` lane
4. 风险卡输出
5. 点击修复后，触发自动修复 + 定向 `ui-ux-review`

先不要做：

- 自动删文件
- 每天全量图谱生成
- 无限制的全仓库重构
