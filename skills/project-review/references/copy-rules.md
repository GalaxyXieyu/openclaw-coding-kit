# Project Review Copy Rules

## 目标

所有项目摘要都要让人一眼看懂，不需要翻任务详情，不需要理解专业术语。

每条摘要尽量回答三件事：

- 做了什么
- 还没做什么
- 下一步做什么

## 字数要求

- 目标：每个项目 `<= 50` 个中文字符
- 超过时优先压缩细节，不要删掉“下一步”

## 推荐模板

优先使用这种一句话结构：

`<项目>做了<已做>，还差<未做>，下一步先<下一步>。`

如果项目名太长，可以省略项目名：

`做了<已做>，还差<未做>，下一步先<下一步>。`

## 允许的表达

这些词更像人话，优先使用：

- 做了
- 改了
- 补了
- 处理了
- 还差
- 还没做
- 卡在
- 下一步
- 先做
- 先看

## 禁止的表达

不要写这些空话、套话、专业黑话：

- 闭环
- 赋能
- 沉淀
- 对齐
- 拉通
- 抓手
- 协同
- 颗粒度
- 方案化
- 业务侧
- 链路
- 驱动
- 抽象能力
- 体系化
- 方法论

## 好例子

- `小程序做了登录页，还差支付联调，下一步先补测试。`
- `官网改了首页，还没动表单，下一步先接提交。`
- `后台补了导出，还差筛选，下一步先修报错。`

## 坏例子

- `小程序链路已完成关键能力沉淀，后续持续闭环推进。`
- `业务侧协同已对齐，下一阶段继续拉通方案化建设。`
- `本项目完成核心能力抽象，后续按照方法论推进。`

## 压缩顺序

超长时按这个顺序压缩：

1. 去掉项目名
2. 缩短已做细节
3. 缩短未做细节
4. 缩短下一步细节

不要删掉“还差/还没做”和“下一步”这两个信息点。
