# PM-Coder-ACP-Bridge 架构背景

生成时间：2026-04-07  
来源：用户口述背景整理

## 1. 这套架构为什么会出现

这套 `pm + coder + acp-progress-bridge` 架构，不是为了做一个“AI 玩具项目”，而是为了解决真实复杂项目里的协作和上下文问题。

原始痛点主要有四类：

### 1.1 单会话 AI 在复杂需求里容易上下文污染

一旦需求稍微复杂，单个 AI 会话很容易出现：

- 上下文越来越脏
- 需求、代码、测试、进度混在一起
- 历史约束被覆盖
- 会话越长越难维护

这会直接降低复杂项目的稳定开发能力。

### 1.2 不同 AI 工具擅长的事情不一样，但原生协作能力不足

用户实际体验是：

- “龙虾”更适合和人沟通、梳理需求、做业务理解
- Codex 更适合在本地做编码、读代码、处理复杂实现
- 但不同 CLI/工具之间原生并行协作并不顺畅

也就是说，真正的瓶颈不是“有没有 AI”，而是“多个 AI 能不能像一个团队一样协作”。

### 1.3 传统需求流转会产生大量信息损耗

原来的开发流程通常是：

1. 业务方提需求
2. PM 理解需求、梳理文档和排期
3. 开发再接手实现
4. 中间反复确认可实现度、边界、优先级

这个链条的问题是：

- 信息在业务 -> PM -> 开发之间不断损耗
- 技术可实现度往往要到后面才暴露
- 来回确认会消耗大量时间
- 需求文档和实际实现很容易脱节

### 1.4 一人公司/小团队非常需要“可随时沟通、可随时开发”的模式

对一人公司或极小团队来说，最重要的不是组织层级，而是：

- 需求能不能快速变成可执行文档
- 客户能不能随时追进度
- 开发能不能实时拿到最新需求
- 系统能不能在复杂项目中依然保持低摩擦

这就是这套架构的现实场景。

## 2. 这套架构想解决什么问题

它本质上是在做一件事：

> 把“需求沟通”和“代码实现”拆成两个 AI 角色，但让它们共享同一个事实源和协作空间。

对应到角色就是：

- PM：负责说人话、理解业务、梳理需求、同步进度、维护项目上下文
- Developer：负责把上下文转换成代码，并通过 ACPX 在本地高质量执行

最终目标不是“多加几个 agent”，而是建立一个稳定的 AI 开发组织模型。

## 3. 为什么是 PM + Developer 这两个核心角色

### 3.1 PM 负责业务语言和上下文稳定

PM 角色的价值不只是“记笔记”，而是：

- 和客户/业务方用自然语言沟通
- 把模糊需求收敛成清晰任务
- 维护共享文档与任务清单
- 实时更新需求变更与项目状态
- 让上下文对所有参与者始终一致

PM 的本质是“稳定上下文入口”。

### 3.2 Developer 负责高质量落地

Developer 角色的价值是：

- 面向代码和实现细节
- 处理复杂工程问题
- 在本地环境中高效开发
- 用更专业的 coding context 完成任务
- 接住 PM 已经整理好的任务和约束

Developer 的本质是“稳定执行出口”。

### 3.3 这两个角色组合后，形成最短路径

以前的路径是：

业务 -> PM -> 开发 -> 反复确认 -> 实现

现在希望变成：

业务/客户 -> PM AI 快速梳理 -> Developer AI 对着稳定需求直接开发

这样做的价值是：

- 减少需求传递损耗
- 更早暴露技术可行性
- 让需求、文档、任务、实现形成闭环
- 沟通和开发可以真正并行

## 4. 为什么核心事实源是“共享云端工作区”

这套架构最关键的基础设施不是某个 agent，而是共享事实源。

这里选择的是飞书工作区，核心包含：

- 文档
- todolist / tasklist

选择它的原因是：

- PM 和 Developer 都能读取同一份项目事实
- 客户/业务方也能在同一空间查看进度与需求
- 新需求、变更、补充说明可以实时进入系统
- 文档和任务不是散落在本地文件，而是共享且持续更新的

这个设计直接解决了“上下文散落”和“需求不同步”的问题。

## 5. 为什么要用 ACPX 串联

ACPX 在这里的作用不是“再加一个技术栈”，而是做执行通道。

它承担的价值包括：

- 把 PM 侧整理好的任务交给 coder 侧执行
- 让执行会话可以独立运转，减少主会话污染
- 让本地开发能力和外部沟通能力连接起来
- 给多角色协同提供统一的运行机制

所以 ACPX 是角色协作的桥，而不是单纯的调度器。

## 6. 为什么 OpenClaw + 飞书 + Codex 这个组合成立

这套组合不是随意拼接，而是各自承担不同职责：

### 6.1 OpenClaw

负责把 AI 能力放到真实协作环境里，尤其是飞书场景中。

### 6.2 飞书

负责承载共享文档、任务、沟通和进度透明化。

### 6.3 Codex

负责本地高质量 coding execution。

它的优势在于：

- 本地代码上下文能力强
- 读代码、改代码、调试更直接
- 自带很多本地能力和 skills，不用再额外搭复杂环境

### 6.4 PM / 龙虾侧

负责更自然的人话沟通、需求吸收、项目管理和客户交互。

组合起来之后形成的能力是：

- 用飞书承载共享事实
- 用 PM 角色做业务沟通和上下文整理
- 用 Codex 做本地高质量开发
- 用 ACPX 和 bridge 把两端串起来

这就是这套架构的核心组合逻辑。

## 7. 为什么还需要 progress bridge

如果只有 PM 和 coder，会有一个问题：

> 执行发生在子会话/本地，但外部沟通侧不知道执行到哪里了。

所以 `acp-progress-bridge` 的价值是：

- 把子会话进度回推给父会话
- 把完成结果做简洁回传
- 避免外部会话为了追进度不断深挖执行细节
- 减少信息爆炸和上下文膨胀

它不是附属功能，而是“让双角色协作可持续”的关键补丁。

### 7.1 自动汇报机制的真实价值

这套 bridge 的意义，不只是“自动发一句完成了”，而是建立一个稳定的外部可见执行回路：

1. PM / coder 把工作派发为一个 Codex ACP 子会话
2. 子会话在本地独立执行，避免主会话被实现细节污染
3. bridge 持续观察子会话的 progress / done / assistant output
4. bridge 把这些内部状态压缩成一条内部更新消息，送回父会话
5. 父会话再把它翻译成对用户可见的自然语言进度或完成总结

这样带来的效果是：

- 外部沟通侧随时知道开发进展
- 执行侧不需要把所有细节直接暴露给主会话
- 进度和结果可以稳定回流到飞书群、共享文档和任务流程
- 可以在多个群、多个会话中并行沟通，但最终仍然回到统一的项目事实源

### 7.2 为什么这对飞书工作区特别重要

因为这套模式不是只想“让 agent 自己知道自己做完了”，而是想让：

- 客户能看见进度
- PM 能追踪状态
- 业务内容能持续回落到文档和 todolist
- 多个群里讨论的内容不会失控散落

所以从产品角度看，`acp-progress-bridge` 不是纯技术插件，而是“共享工作区闭环”的关键部件。

## 8. 这套架构适合什么项目

这套模式不是为简单 demo 设计的。

用户给出的验证案例是：

- 宠物展示
- 后台管理
- 支付
- 新手指引
- CI/CD

这说明它已经被用于相对复杂、带多模块、多流程、多角色协作需求的项目，而不是只适合：

- todolist
- 贪吃蛇
- 单页 demo

所以这套架构的定位应该是：

> 面向真实复杂项目的 AI 协作开发模式。

## 9. 这套架构的核心设计原则

从背景上看，这套架构至少有 6 个明确原则：

### 9.1 需求和实现分角色，但不分事实源

角色不同，但上下文必须统一。

### 9.2 沟通体验和开发效率同时优化

不是只追求“代码快”，而是追求整体交付链路更顺。

### 9.3 上下文要可持续维护

不能靠单个长会话硬扛复杂项目。

### 9.4 需求变更必须能实时进入执行系统

客户补充、修改、追进度，不应该脱离开发系统。

### 9.5 要允许并行

PM 与 Developer 不是串行切换，而应并行协作。

### 9.6 要适合一人公司/小团队的真实经营模式

不是模仿大公司流程，而是服务高频变化、强沟通、强交付的小团队现实。

## 10. 对这个仓库的启发

这份背景说明了：

这个仓库真正要做的，不只是提供几个脚本和插件，而是把下面三层东西产品化：

1. 协作方法论
2. 共享上下文基础设施
3. 角色之间的自动化桥接能力

所以后续做 GSD 规划时，不能只围绕“安装、路径、脚本重构”，还要明确三个产品目标：

- 让这套模式更容易安装和复制
- 让这套模式更稳定地支撑复杂项目
- 让不同平台和不同开发者都能接入这套协作方式

## 11. 后续在 PROJECT.md 中建议保留的核心表述

后续如果进入 `gsd-new-project`，建议至少保留这些关键信息：

- 这是一个面向复杂项目的 AI 协作开发架构，不是玩具 demo 工具集
- 核心模式是 PM + Developer 双角色协作
- 共享云端工作区是事实源，飞书文档和任务是核心基础设施
- ACPX 用来串联沟通侧和执行侧
- progress bridge 用来把子会话执行信息压缩回传，避免上下文爆炸
- 目标用户包含一人公司、小团队，以及需要高频需求沟通的项目场景
